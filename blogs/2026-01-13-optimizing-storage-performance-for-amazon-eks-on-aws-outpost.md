<p><a href="https://aws.amazon.com/eks/" rel="noopener noreferrer" target="_blank">Amazon Elastic Kubernetes Service (Amazon EKS)</a> on <a href="https://aws.amazon.com/outposts/" rel="noopener noreferrer" target="_blank">AWS Outposts</a> brings the power of managed <a href="https://docs.aws.amazon.com/eks/latest/userguide/kubernetes-concepts.html" rel="noopener noreferrer" target="_blank">Kubernetes</a> to your on-premises infrastructure. Use Amazon EKS on Outposts rack to create hybrid cloud deployments that maintain consistent AWS experiences across environments. As organizations increasingly adopt edge computing and hybrid architectures, storage optimization and performance tuning become critical for successful workload deployment.</p> 
<p>Outposts extend AWS infrastructure, services, APIs, and tools to virtually any datacenter, co-location space, or on-premises facility. In this blog post you will learn about your storage options and their performance characteristics which is essential for building resilient, high-performing applications using Amazon EKS on Outposts.</p> 
<h2>Amazon EKS on Outposts deployment options</h2> 
<p>The following two sections outline the differences between <a href="https://docs.aws.amazon.com/eks/latest/userguide/eks-outposts.html" rel="noopener noreferrer" target="_blank">Amazon EKS extended and local cluster deployment options available on Outposts</a>.</p> 
<h3>Amazon EKS extended cluster architecture</h3> 
<p>Amazon EKS extended clusters on Outposts provide a powerful solution for organizations seeking to use the benefits of Kubernetes while maintaining certain workloads on-premises, as shown in the following figure. This hybrid architecture allows businesses to extend their EKS clusters from the AWS Cloud to their own data centers or edge locations using Outposts. The Kubernetes control plane remains in the <a href="https://aws.amazon.com/about-aws/global-infrastructure/regions_az/" rel="noopener noreferrer" target="_blank">AWS Region</a>, providing centralized management and benefiting from the AWS infrastructure in the cloud and on the Outpost.</p> 
<p>Outposts is designed to be a connected service, and needs reliable<strong>&nbsp;</strong>network connectivity to the AWS Region using the <a href="https://docs.aws.amazon.com/outposts/latest/userguide/service-links.html" rel="noopener noreferrer" target="_blank">Outposts service link</a>.</p>

Having the Kubernetes control plane hosted on your Outpost also reduces latency for cluster operations.</p>

<h2>Storage options for Amazon EKS extended clusters on Outposts</h2> 
<p><a href="https://kubernetes.io/docs/concepts/storage/persistent-volumes/" rel="noopener noreferrer" target="_blank">Persistent Volumes (PV)</a> and <a href="https://kubernetes.io/docs/concepts/storage/persistent-volumes/" rel="noopener noreferrer" target="_blank">Persistent Volume Claims (PVC)</a> serve as a critical abstraction layer in Kubernetes, separating the storage consumption details from storage provisioning, and allowing administrators to manage storage resources independently from how applications consume them. PVs and PVCs make sure of data persistence across pod restarts and rescheduling events, making them essential for applications that need to maintain state, such as databases, file storage systems, and other data-intensive workloads. The abstraction provided by PV and PVC enables platform-agnostic storage management, where applications can request storage through PVCs without needing to know the underlying storage implementation details. PVs and PVCs support dynamic provisioning through <a href="https://kubernetes.io/docs/concepts/storage/storage-classes/" rel="noopener noreferrer" target="_blank">Storage Classes</a>, allowing for automated storage allocation based on application demands, while also providing features such as access modes, capacity management, and reclaim policies to effectively manage the storage lifecycle in a Kubernetes cluster.</p> 
<h3>Integrating Amazon EBS with Amazon EKS</h3> 
<p><a href="https://aws.amazon.com/ebs/" rel="noopener noreferrer" target="_blank">Amazon Elastic Block Store (Amazon EBS)</a> provides high-performance block storage that’s ideal for low-latency applications providing consistent performance. When deployed on Outposts racks, EBS volumes are stored on the Outposts hardware, providing significant performance advantages over network-attached storage solutions, as shown in the following figure.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/01/06/pic3.png"><img alt="" class="aligncenter wp-image-25556 size-full" height="1961" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/01/06/pic3.png" width="1430" /></a></p> 
<p style="text-align: center;">Figure 3 : Integrating Amazon EBS with Amazon EKS on Outposts</p> 
<h3>Benefits and use cases</h3> 
<ul> 
 <li><strong>Storage</strong>: EBS volumes on Outposts racks provide data access without dependency on external connectivity.</li> 
 <li><strong>Performance:</strong> Local storage delivers consistent latency and high IOPS/throughput.</li> 
 <li><strong>Cost: </strong>On-premises storage eliminates data transfer costs and reduces bandwidth needs, lowering the total cost of ownership.</li> 
</ul> 
<h3>Implementation considerations</h3> 
<p>Consider the following when using EBS on Outposts rack:</p> 
<ul> 
 <li>EBS volumes on Outposts are tied to a single rack and the availability zone the Outpost is homed to, needing applications to address single-point-of-failure risks.</li> 
 <li>Protect data using EBS snapshots in the parent Region and schedule regular backups.</li> 
 <li>Capacity on Outposts is finite, monitor Outposts storage usage and plan expansions proactively to avoid insufficient capacity errors.</li> 
</ul> 
<p>Refer to <a href="https://github.com/kubernetes-sigs/aws-ebs-csi-driver/tree/master/examples/kubernetes/dynamic-provisioning" rel="noopener noreferrer" target="_blank">Dynamic Volume Provisioning</a>&nbsp;to learn more about deploying pod with the EBS volume attached.</p> 
<h3>Amazon EFS with Amazon EKS</h3> 
<p><a href="https://aws.amazon.com/efs/" rel="noopener noreferrer" target="_blank">Amazon Elastic File System (Amazon EFS)</a> provides scalable, shared file storage that can be accessed across multiple AWS Availability Zones (AZs) and on-premises environments. Although Amazon EFS with Amazon EKS on Outposts maintains the same setup procedures as standard cloud deployments, there is a critical dependency on the <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/service-links.html" rel="noopener noreferrer" target="_blank">service link</a> connection between your Outposts and the AWS Region. Amazon EFS is not a locally supported service on Outposts, so connectivity to the AWS Region is required to use this service with your Outpost.</p> 
<p>Amazon EFS allows multiple pods to concurrently access shared file systems. It is well-suited for applications that need collaborative data access, content management, and distributed processing workloads.</p> 
<h4>Amazon EFS as a persistent storage solution for Amazon EKS extended cluster instances</h4> 
<p>Amazon EFS as a PV for your Amazon EKS extended cluster operates through a hybrid architecture where the Amazon EFS file system resides in the Region, but mount points can be created on the worker nodes running on Outposts subnets through the service link as shown in the following figure.</p>

<p style="text-align: center;">Figure 4 : Amazon EFS as a persistent storage solution for extended clusters</p> 
<h4>Benefits and use cases</h4> 
<ul> 
 <li><strong>Shared storage capabilities</strong>: multiple pods can access a centralized file system, enabling shared data, code, and assets across instances.</li> 
 <li><strong>Scalability</strong>: storage capacity and performance automatically scale with usage, eliminating manual provisioning and upfront planning.</li> 
 <li><strong>Compliance</strong>: Amazon EFS provides full file system features and compatibility for traditional applications, such as locking, permissions, and directory structure.</li> 
</ul> 
<h4>Challenges and limitations</h4> 
<p>Consider the following when using Amazon EFS with Outposts:</p> 
<ul> 
 <li>Network latency: file access involves network traversal to Amazon EFS in the Region, adding more latency and making small or metadata operations potentially slow for latency-sensitive applications.</li> 
 <li>Throughput: aggregate throughput is restricted by the available bandwidth on the service link between the Outposts and AWS Region. This impacts concurrent access and large file transfers during peak usage.</li> 
 <li>Dependency on AWS Region connectivity: Amazon EFS needs continuous connectivity to the parent Region. Disruptions may affect file system availability, operations, and disaster recovery processes.</li> 
 <li>Data Transfer charges: Since EFS is in AWS Parent region and EKS worker nodes and pods are in Outpost additional charges are applicable.</li> 
</ul>

<h2>Best practices for optimizing performance</h2> 
<p>Optimizing performance starts with selecting the right storage type for your workload: Amazon EBS for low-latency, high-throughput block storage; Amazon EFS for shared POSIX-compliant file systems; and Amazon S3 for scalable object storage with API compatibility. Ensure proper volume sizing, monitor usage proactively, and configure CPU and memory requests accurately to balance performance and efficiency—auto scaling and QoS classes can further optimize resource management. Improve data locality by using local storage, apply caching with intelligent eviction, and design for efficient, asynchronous, and compressed data access patterns.</p> 
<h2>Monitoring and observability</h2> 
<p>Monitoring key performance metrics is essential to maintain storage efficiency and application reliability. For Amazon EBS, track IOPS, throughput, latency, burst balance, queue depth, and snapshot performance to avoid degradation—see the <a href="https://docs.aws.amazon.com/outposts/latest/userguide/outposts-cloudwatch-metrics.html#metrics-ebs" rel="noopener noreferrer" target="_blank">Amazon CloudWatch metrics for Amazon EBS</a> for the full list. For Amazon EFS, monitor total I/O, throughput, client connections, metadata operations, burst credits, and Regional data transfers to support effective capacity planning—refer to <a href="https://docs.aws.amazon.com/efs/latest/ug/efs-metrics.html" rel="noopener noreferrer" target="_blank">CloudWatch metrics for Amazon EFS</a>. For Amazon S3, observe request and error rates, data transfer, storage usage, latency, multipart upload efficiency, and access patterns to optimize performance and cost—see <a href="https://docs.aws.amazon.com/AmazonS3/latest/userguide/metrics-dimensions.html" rel="noopener noreferrer" target="_blank">Metrics and dimensions</a>.</p> 
<h2>Security considerations</h2> 
<p>Strong security practices are critical for Amazon EKS on Outposts. Use <a href="https://aws.amazon.com/kms/" rel="noopener noreferrer" target="_blank">AWS Key Management Service (AWS KMS)</a> for Amazon EBS encryption, encrypt Amazon EFS data at rest and in transit, and enable server- or client-side encryption for Amazon S3. Enforce TLS for all data transfers and apply key rotation with compliance controls. Implement least privilege IAM policies, scoped roles, and Kubernetes Role-Based Access Control (RBAC) for granular pod access. Secure traffic with security groups and NACLs, and maintain audit logs for all storage operations.</p> 
<h2>Cost optimization strategies</h2> 
<p>Manage storage costs by right-sizing volumes, automating lifecycle policies, selecting appropriate storage classes, monitoring data transfer, and using de-duplication and compression where applicable. Lower operational expenses through automated backups, infrastructure as code (IaC), monitoring automation, leveraging managed services, applying cost allocation tags, and conducting regular usage reviews.</p>

By selecting the right storage solution for each workload and leveraging Outposts’ local infrastructure, you can reduce latency, minimize network dependencies, and maintain consistency across environments. As Outposts capabilities continue to evolve, they offer a strong foundation for modern, resilient, and cost-efficient hybrid cloud architectures.</p>
