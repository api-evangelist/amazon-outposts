<p><em>This post is written by Adam Imeson, Sr. Hybrid Edge Specialist Solutions Architect.</em></p> 
<p><a href="https://aws.amazon.com/outposts/rack/">AWS Outposts rack</a> is a fully-managed service that offers the same AWS infrastructure, APIs, tools, and a subset of AWS services to any data center, colocation space, or on-premises facility for a consistent hybrid experience. Outposts rack is ideal for workloads that require low latency, access to on-premises systems, local data processing, data residency, and migration of applications with local system interdependencies.</p> 
<p><a href="https://docs.aws.amazon.com/outposts/latest/userguide/what-is-outposts.html">An Outpost</a> is a pool of AWS compute and storage capacity deployed at a customer site. In an Outposts rack deployment, an Outpost may comprise of one or more racks connected together at the site. It’s common for customers to order their Outpost in a dedicated account and then integrate with their multi-account organizational architecture by sharing the Outpost via <a href="https://aws.amazon.com/ram/">AWS Resource Access Manager (AWS RAM)</a>. This post will explain how to set up cross-account <a href="https://aws.amazon.com/cloudwatch/">Amazon CloudWatch</a> metrics so that disparate stakeholders within your organization can effectively monitor your Outpost’s capacity to meet their specific needs.</p> 
<h2>Overview</h2> 
<p>The AWS account that you use to order an Outpost owns that Outpost. This includes all metrics and health events pertaining to that Outpost. Many customers must integrate Outposts into their multi-account environments, as discussed in the “Best practices: AWS Outposts in a multi-account AWS environment” posts (<a href="https://aws.amazon.com/blogs/mt/best-practices-aws-outposts-in-a-multi-account-aws-environment-part-1/">part 1</a> and <a href="https://aws.amazon.com/blogs/mt/best-practices-aws-outposts-in-a-multi-account-aws-environment-part-2/">part 2</a>). This post will go into more detail on how to monitor Outposts in these environments.</p> 
<p>The nuance here stems from the different ways to share access to AWS resources. AWS RAM allows infrastructure resources to be shared across multiple accounts. Then, the consumer accounts can launch resources on the infrastructure as though they owned it. <a href="https://aws.amazon.com/iam/">AWS Identity and Access Management (IAM)</a> allows customers to modify a given account’s permissions such that users in other accounts can make AWS API calls that affect the given account.</p> 
<p>An Outpost provides infrastructure resources, so customers can share Outposts via AWS RAM. CloudWatch metrics <em>about</em> Outposts are data which customers retrieve using AWS API calls, so customers can share access to those metrics using IAM.</p> 
<p>In a typical customer’s <a href="https://aws.amazon.com/organizations/">AWS Organization</a>, there are two cases to consider. First, when the customer is sharing an Outpost to multiple development accounts, each account needs to view metrics relevant to the Outpost so that the development accounts can deploy and operate their applications.<a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture01.png"><img alt="Diagram depicting an Outpost that a customer has shared to three different accounts using RAM. The three different accounts each have a different application deployed in them." class="aligncenter size-full wp-image-19473" height="579" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture01.png" width="827" /></a></p> 
<p>Second, when the customer has several accounts that each own different Outposts, the customer’s centralized monitoring account needs to track metrics relevant to each of the Outposts.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture02.png"><img alt="Diagram depicting three accounts that each own a separate Outpost, with all three accounts sharing Outpost metrics to CloudWatch in the customer’s central monitoring account." class="aligncenter size-full wp-image-19474" height="650" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture02.png" width="827" /></a></p> 
<p>This post will explain strategies for both cases.</p> 
<p>Customers must monitor the health of the Outpost’s connection to its regional control plane (the Outpost’s <a href="https://docs.aws.amazon.com/outposts/latest/userguide/how-outposts-works.html#region-connectivity">service link</a>), as an Outpost is an extension of an <a href="https://aws.amazon.com/about-aws/global-infrastructure/regions_az/">AWS Availability Zone</a> (AZ) and is designed to be connected to an AZ at all times. The health of the Outpost’s service link is a crucial variable when application owners are diagnosing disruptions to their application, and also when infrastructure owners are diagnosing disruptions to a site.&nbsp;Customers can monitor their service link’s status with the <a href="https://docs.aws.amazon.com/outposts/latest/userguide/outposts-cloudwatch-metrics.html#outposts-metrics">ConnectedStatus</a> metric.</p> 
<p>Customers also must monitor their Outposts’ current capacity. Outposts necessarily have a limited capacity footprint when compared to an AWS Region. Application owners must make informed decisions about capacity as they scale their apps over time or respond to occasional hardware failures. Infrastructure owners also must maintain a holistic view of capacity across all of the Outposts for which they are responsible so that they can plan for capacity expansion over time.&nbsp;Customers can monitor their Outposts’ capacity using the various <a href="https://docs.aws.amazon.com/outposts/latest/userguide/outposts-cloudwatch-metrics.html#outposts-metrics">capacity metrics</a> that Outposts provide.</p> 
<p>For an overview of how to set up a capacity dashboard and capacity-based CloudWatch alarms within a single account, see&nbsp;<a href="https://aws.amazon.com/blogs/compute/monitoring-aws-outposts-capacity/">“Monitoring AWS Outposts capacity.”</a> This post will expand on the single-account strategy by introducing cross-account capabilities. See also <a href="https://aws.amazon.com/blogs/aws/cross-account-cross-region-dashboards-with-amazon-cloudwatch/">“Cross-Account Cross-Region Dashboards with Amazon CloudWatch.”</a> These two posts provide practical walkthroughs for setting up the metric flows explained below.</p> 
<h2>Setting up Outposts metric permissions for your organization</h2> 
<p>This post assumes that you have multiple Outposts in different accounts that are all part of the same Organization. You’re sharing these Outposts into accounts that development teams use to deploy and operate their applications. You also have a centralized monitoring account where your infrastructure team tracks various metrics across all accounts. Your Organization might look something like this:</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture03.png"><img alt="A base diagram depicting six AWS accounts with different names. Outpost Account 1 contains an Outpost. Outpost Account 2 contains a different Outpost. Monitoring Account contains Amazon CloudWatch. Accounts A through C contain Applications A through C respectively." class="aligncenter size-full wp-image-19475" height="497" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture03.png" width="772" /></a></p> 
<p>The first Outpost is shared to Accounts A and B, and the second Outpost is only shared to Account B. This is just an example of how a customer might set up their environment so that Application A can deploy on Outpost 1, and Application B can deploy on both Outpost 1 and 2.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture04.png"><img alt="The same base diagram of the six AWS accounts as before, with arrows added to depict AWS RAM resource shares. <p>To enable centralized monitoring, each account shares CloudWatch metrics with the central monitoring account as described in <a href="https://aws.amazon.com/blogs/aws/cross-account-cross-region-dashboards-with-amazon-cloudwatch/">“Cross-Account Cross-Region Dashboards with Amazon CloudWatch.”</a></p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture05.png"><img alt="The same base diagram of the six AWS accounts, with arrows added to depict CloudWatch metrics being shared from all five of the other accounts to the Monitoring Account." class="aligncenter size-full wp-image-19478" height="497" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture05.png" width="772" /></a></p> 
<p>Now there are application accounts which can launch on the desired Outposts, and all of the accounts are sharing metrics with the central monitoring account. The team responsible for procuring and managing the Outposts can now set up dashboards in the central monitoring account in accordance with&nbsp;<a href="https://aws.amazon.com/blogs/compute/monitoring-aws-outposts-capacity/">“Monitoring AWS Outposts capacity”</a> to get a holistic view of capacity. This is valuable for capacity planning as applications naturally grow over time.</p> 
<p>However, this may not be sufficient for operations. Consider that each application team needs to understand how much capacity is available on the Outpost that they’re using. This is crucial for teams operating highly available applications to maintain awareness of whether they still have N+1 capacity available on the Outpost to use in the event of a hardware failure. This is also important for planning expansions to the application ahead of time, as application teams have the best understanding of the future needs of their applications. Finally, application teams can use the metrics to track the operational health of the Outpost, which is crucial for root-causing any application disruptions.</p> 
<p>You can implement this by sharing CloudWatch metrics from the Outpost accounts to the application accounts which are consuming the Outposts’ capacity, as shown in the following diagram.</p>

<h2>Walkthrough</h2> 
<p>Log in to your application account and navigate to the <strong>CloudWatch</strong> console. Open the <strong>Settings</strong> menu and choose <strong>Configure</strong>.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture07.png"><img alt="Screenshot of the CloudWatch Console’s Settings menu." class="aligncenter size-large wp-image-19480" height="187" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture07-1024x187.png" width="1024" /></a></p> 
<p>Scroll to the bottom. In the <strong>View cross-account cross-region</strong> section, choose <strong>Edit</strong>.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture08.png"><img alt="Screenshot of the Cross-account cross-region sub-menu in the CloudWatch console." class="aligncenter size-full wp-image-19481" height="447" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture08.png" width="977" /></a></p> 
<p>Choose your preferred account selection method from the three options and choose <strong>Save changes</strong>. I recommend the <strong>Custom account selector</strong> option, as it strikes a good balance between a simple setup and ease of use. If you choose this option, then input the Outpost owner account’s account ID and a human-readable name for the account. This name will appear in the drop-down when you’re using the <strong>CloudWatch</strong> console to view metrics from other accounts later.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture09.png"><img alt="Screenshot of the Cross-account cross-region sub-menu in the CloudWatch console, with the “Custom account selector” option selected and “123456789012 Outpost owner account” in the input field." class="aligncenter size-large wp-image-19482" height="469" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture09-1024x469.png" width="1024" /></a></p> 
<p>Your application account is now prepared to view metrics from the Outpost owner account. Now log in to the account that owns the Outpost and navigate to the <strong>CloudWatch</strong> console. You still need to share the Outpost’s metrics to the application account. Open the <strong>Settings</strong> page again, and choose <strong>Configure</strong> in the <strong>Cross-account cross-region </strong>section as before. This time, choose <strong>Share data</strong> in the <strong>Share your CloudWatch data</strong> section:</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture10-3.png"><img alt="Screenshot of the Cross-account cross-region sub-menu in the CloudWatch console, with the “Share data” button circled in red in the “Share your CloudWatch data” section." class="aligncenter size-large wp-image-19483" height="469" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture10-3-1024x469.png" width="1024" /></a></p> 
<p>Choose <strong>Add account</strong> and input the application account’s account ID. Then scroll to the bottom of the page and choose <strong>Launch CloudFormation template</strong>.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture11-1.png"><img alt="Screenshot of the “Share your CloudWatch data sub-menu in the CloudWatch console. The “Specific accounts” option in the “Sharing” section is highlighted, and the sample account ID “234567890123” is typed into the input field." class="aligncenter size-large wp-image-19484" height="527" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture11-1-1024x527.png" width="1024" /></a></p> 
<p>The <a href="https://aws.amazon.com/cloudformation/">AWS CloudFormation</a> template will create the <strong>CloudWatch-CrossAccountSharingRole</strong>. This role gives CloudWatch read access to the AWS account that you specified, the application account. You can view and modify this role using the IAM console if you want to. For example, you might adjust the role to allow read access to an entire Organizational Unit (OU).</p> 
<p>Now, log back in to the application account and navigate to the <strong>CloudWatch</strong> console. Choose <strong>All metrics</strong> in the left-side menu. In the <strong>Metrics</strong> section, select the <strong>Outpost owner account</strong> from the drop-down.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture12-1.png"><img alt="Screenshot of the CloudWatch console’s “All metrics” sub-page. The account selection drop-down is circled in red in the “Metrics” subsection." class="aligncenter size-large wp-image-19485" height="401" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2022/11/28/Picture12-1-1024x401.png" width="1024" /></a></p> 
<p>You can now view the metrics from the Outpost owner account and incorporate them into the dashboards in the application account. Now the application teams can track the Outposts’ <code>ConnectedStatus</code> metrics to be alerted on any disconnections from the region, and they can track the Outposts’ capacity metrics as well. It’s a best practice to alarm on Outpost capacity metrics once a consumption threshold defined by business needs has been breached.</p> 
<h2>Conclusion</h2> 
<p>Outposts rack allows customers to deploy AWS infrastructure into virtually any data center, colocation space, or on-premises facility. Outposts are tied to the AWS account that ordered them, and customers can share Outposts among AWS accounts within the same Organization. When multiple teams within a customer’s Organization are interacting with the same Outpost, that introduces additional monitoring surface area for capacity and service health. This post explains how customers can accommodate their teams’ different needs by sharing Outposts metrics around their Organization along with their Outposts. As best practices, customers should share their Outposts capacity and <code>ConnectedStatus</code> metrics to teams who are running applications on Outposts. Customers’ operations teams should also work with their stakeholders to define a maximum capacity utilization threshold for a given Outpost and alarm on that threshold.</p>
