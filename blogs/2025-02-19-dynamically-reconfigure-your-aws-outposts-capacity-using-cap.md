<p><em>This post is written by Brianna Rosentrater, Hybrid Edge Specialist SA and Adam Duffield, Senior Technical Account Manager.</em></p> 
<p><a href="https://aws.amazon.com/outposts/" rel="noopener" target="_blank">AWS Outposts</a> extends AWS infrastructure, AWS services, APIs, and tools to on-premises locations for workloads that require low latency, local data processing, or data residency. Outposts comes in a variety of form factors, from 42U <a href="https://aws.amazon.com/outposts/rack/?nc=sn&amp;loc=2" rel="noopener" target="_blank">Outposts racks</a> to 1U and 2U <a href="https://aws.amazon.com/outposts/servers/?nc=sn&amp;loc=3" rel="noopener" target="_blank">Outposts servers</a>. Outposts now supports self-service capacity management, making it easy for you to view and manage compute capacity on your Outposts. A default capacity configuration for each new Outpost is determined during the ordering process. This default configuration can subsequently be modified to create a range of instance sizes and quantities to meet your changing business needs. To do so, you create a capacity task, specify the instance sizes and quantity, and run the capacity task to implement the changes. This post focuses on how to use capacity tasks to perform multi-host reconfigurations and view existing capacity configurations.</p> 
<h2>Overview</h2> 
<p>Amazon <a href="https://aws.amazon.com/ec2/" rel="noopener" target="_blank">Elastic Compute Cloud (Amazon EC2)</a> capacity on an Outpost is determined by the total volume of compute capacity within the Outpost when ordered. Outposts can also be scaled up or out as needed during your commitment term. For further details on Outpost capacity planning including best practices, refer to the <a href="https://docs.aws.amazon.com/whitepapers/latest/aws-outposts-high-availability-design/capacity-planning.html" rel="noopener" target="_blank">Capacity Planning – AWS Outposts High Availability Design and Architecture whitepaper</a>. We recommend planning spare capacity for N+M host availability when making modifications to your Outpost capacity configuration if your workloads need to be highly available. To calculate, take the number of hosts (N) you need to run all your workloads, and then add (M) additional hosts to meet your requirements for server availability during failure and maintenance events.</p> 
<h2>View existing Outposts capacity configuration</h2> 
<p>Outposts users now have visibility into capacity configurations at both an instance family and host level. This gives greater insight into capacity usage and instance placements. Within the Outposts console, after choosing the Outpost ID on which you want to view the capacity configuration, two new views have been provided: the <strong>Instance view</strong> and the <strong>Rack view</strong>.</p> 
<h3>Instance view</h3> 
<p>The <strong>Instance view</strong> provides a granular breakdown of the currently deployed instances on the Outpost along with an overall view of instance family capacity pools and their usage, as shown in the following figure. The <strong>Instances</strong> section gives detailed information around the deployed instances, their associated instance ID, instance size, AWS managed service (if applicable), and asset ID of where the instance is running.</p>

Each host of the same instance family contributes its capacity to the overall pool, which is represented in this section as a percentage rather than number of instance slots. This shows the configured capacity, but it doesn’t reflect any level of usage.</p>

<h3>Rack view</h3> 
<p>The <strong>Rack view</strong> tab provides a more granular view of the overall configuration of each host on a per rack basis, as shown in the following figure. It allows you to analyze the spread and usage of the instance size allocations across each host (asset) and, when choosing the <strong>show instance details</strong> button, provides the instance ID of each used slot. Using the search box, you can filter by <strong>Instance Family</strong> or <strong>Instance Size</strong> to provide a more concise view. If you’re using Outposts server, the Rack view tab will show the capacity configuration of your server.</p>

<h3>Obtaining a view of current configuration</h3> 
<p>Alongside these views, two buttons are available on each of the pages. The <strong>Export JSON</strong> button gives the ability to download a JSON formatted copy of the current configuration for an Outpost. This is especially useful if you’re looking to record current state, or wanting to use the JSON upload option when submitting a new capacity task. The JSON file structure provides the overall configuration of each capacity pool. However, it doesn’t provide any details in terms of usage. The second button, <strong>Modify Instance Capacity</strong>, provides a shortcut to creating a capacity task.</p> 
<p>This level of capacity visibility is also now available through <a href="https://docs.aws.amazon.com/cli/latest/reference/outposts/#cli-aws-outposts" rel="noopener" target="_blank">AWS Command Line Interface (AWS CLI)</a>/<a href="https://docs.aws.amazon.com/outposts/latest/APIReference/API_Operations.html" rel="noopener" target="_blank">Outposts API</a> calls, which some may prefer over the console. For example, the <code>list-assets</code> CLI command can be used to obtain a breakdown of the capacity configuration of each Outpost host:</p> 
<p><code>aws outposts list-assets --outpost-identifier <em>outpost-arn</em></code></p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-5-list-assets-CLI-command-sample-output.png"><img alt="Figure 5 - list-assets CLI command sample output" class="size-full wp-image-23287 aligncenter" height="494" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-5-list-assets-CLI-command-sample-output.png" width="413" /></a></p> 
<p style="text-align: center;"><em>Figure 5 – list-assets CLI command sample output</em></p> 
<p>If you want to obtain details of running instances on an Outpost, such as the instance size, the asset ID on which an instance is running, and the related AWS service name (if relevant), then the <code>list-asset-instances</code> CLI command can be used:</p> 
<p><code>aws outposts list-asset-instances --outpost-identifier <em>outpost-arn</em></code></p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-6-list-asset-instances-CLI-command-sample-output.png"><img alt="Figure 6 - list-asset-instances CLI command sample output" class="size-full wp-image-23286 aligncenter" height="566" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-6-list-asset-instances-CLI-command-sample-output.png" width="358" /></a></p> 
<p style="text-align: center;"><em>Figure 6 – list-asset-instances CLI command sample output</em></p> 
<p>The <code>list-asset-instances</code> CLI command also allows you to filter through numerous dimensions, such as instance type or AWS service. For example, this can be particularly useful for quickly identifying all running instances of a certain type, such as the m5d.large instance type by using the following command:</p> 
<p><code>aws outposts list-asset-instances --outpost-identifier <em>outpost-arn</em> --instance-type-filter m5d.large</code></p> 
<h2>Modifying the Outposts capacity configuration</h2> 
<p>Due to the finite nature of Outposts capacity, changing operational requirements often mean that adjustments need to be made to your Outposts capacity configuration over time as new workloads are identified or applications need scaling.</p> 
<p>From the Outposts console page, choosing <strong>Capacity Tasks</strong> from the left-hand menu gives a list of previously run capacity tasks and their status. From here, you can choose <strong>Create Capacity Task</strong> to start the process. To create a capacity task there are two options available: using an interactive capacity configuration tool through the <strong>Modify an Outpost capacity configuration</strong> option, or uploading a JSON file containing the necessary configuration through the <strong>Upload a capacity</strong> <strong>configuration</strong> option.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-7-Capacity-Tasks-web-form.png"><img alt="Figure 7 - Capacity Tasks web form" class="size-full wp-image-23285 aligncenter" height="1190" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-7-Capacity-Tasks-web-form.png" width="2076" /></a></p> 
<p style="text-align: center;"><em>Figure 7 – Capacity Tasks web form</em></p> 
<p>The interactive <strong>Modify an Outpost capacity configuration</strong> option using the simple <strong>Auto-balance </strong>feature and UI is the easiest way for those unfamiliar with Outpost capacity management to get started with making changes. Using this option, you can also choose one of two methods for the task:</p> 
<ul> 
 <li><strong>Run once:</strong> This results in the capacity task attempting to run a single time. If any instances block the successful application of the configuration, then the task fails.</li> 
 <li><strong>Run periodically over 48 hours or less</strong>: In the event of blocking instances, the capacity task is paused until the instances are stopped. The task rechecks the status every 10 minutes until it can run. If instances aren’t stopped within 48 hours, then the task is cancelled.</li> 
</ul> 
<p>To build out the capacity task, capacity pools are displayed, grouped by instance families, and automatically populated with the current configuration of instance sizes and corresponding vCPU allocation. From here, you can make necessary changes to the existing capacity. Specifically, you can add new instance sizes and amend instance quantities in the corresponding fields. The total vCPU count for each instance family will update automatically to reflect your changes. The <strong>Auto-balance</strong> allows you to automatically adjust the quantities of individual instance sizes to fit within the total vCPU capacity available for the host, which is reflected for each capacity pool at the end. In the event that a capacity pool is over- or under-used, a warning is displayed. You can under provision a host if you choose, but the unprovisioned capacity is unusable, and attempting to overprovision the host results in the capacity task failing to run.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-8-Modifying-existing-capacity-configuration.png"><img alt="Figure 8 - Modifying existing capacity configuration" class="size-full wp-image-23284 aligncenter" height="868" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-8-Modifying-existing-capacity-configuration.png" width="2060" /></a></p> 
<p style="text-align: center;"><em>Figure 8 – Modifying existing capacity configuration</em></p> 
<p>When the necessary changes have been made to the capacity pools, the second part of the capacity task configuration is choosing instances that should not be impacted by the running of the capacity task. During the run, you may not be in a position to stop certain instances, such as databases or AWS managed services such as <a href="https://aws.amazon.com/elasticloadbalancing/" rel="noopener" target="_blank">Elastic Load Balancing (ELB)</a> or <a href="https://aws.amazon.com/pm/elasticache/?gclid=Cj0KCQiA0--6BhCBARIsADYqyL8da4ZviD8nODX8nPJqgy0IwGOVGW52FWsRyUAdVmNNxQW2kGe3u4saApCvEALw_wcB&amp;trk=3ca6c203-af44-4583-82f8-d4911238044b&amp;sc_channel=ps&amp;ef_id=Cj0KCQiA0--6BhCBARIsADYqyL8da4ZviD8nODX8nPJqgy0IwGOVGW52FWsRyUAdVmNNxQW2kGe3u4saApCvEALw_wcB:G:s&amp;s_kwcid=AL!4422!3!673938861344!e!!g!!amazon%20elasticache!19852662212!156481686234" rel="noopener" target="_blank">Amazon ElastiCache</a>, due to the impact to production workloads. Choosing these instances allows the capacity task to automatically try to find a path that avoids impacting them. However, in some situations capacity tasks may fail if the chosen instances block the successful running of the task, and there is no possible solution that avoids all the chosen instances. For example, if a capacity task was to remove all c5.xlarge instances and an instance was chosen to ‘keep as-is’ that was running on this instance size, then the task would fail. To avoid this, make sure to include these instances in your capacity configuration. For example, if you have five critical m5.4xlarge instances that must remain running, then include 5 m5.4xlarge instances in your m5 capacity pool configuration.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-9-Instances-to-keep-as-is.png"><img alt="Figure 9 - Instances to keep as-is" class="size-full wp-image-23283 aligncenter" height="884" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-9-Instances-to-keep-as-is.png" width="2070" /></a></p> 
<p style="text-align: center;"><em>Figure 9 – Instances to keep as-is</em></p> 
<p>After configuring the capacity pools and choosing the necessary instances to keep as-is, an overview of the changes is presented allowing you to validate the configuration prior to running. When you have reviewed the summary, select <strong>Create Task</strong> to trigger the execution of the capacity task using the chosen method. You can observe the status of a capacity task by choosing the capacity task ID. When it’s initially submitted, the status shows as <strong>Requested</strong>. During this time, the capacity task evaluates the necessary changes to determine if the task can proceed or if instances need stopping. If the <strong>Run once</strong> option was chosen and instances do need stopping, then the task moves to a <strong>cancelled</strong> status and provides details of the blocking instances. Alternatively, if <strong>Run periodically</strong> was chosen, the task remains in the <strong>Requested</strong> status until the listed instances have been stopped. In the event that the blocking instances can’t be stopped, the capacity task is cancelled after 48 hours. While a task is running, the Outpost hosts that are impacted by the configuration changes are placed into an isolated state. This means that capacity for new instance launches may be impacted. This isolation only lasts a few minutes while the capacity task is running, but it may impact auto scaling groups if a capacity task coincides with a scaling event.</p> 
<p>Instead of using the interactive capacity configurator UI, you can also choose to upload a JSON file to the console containing the necessary configuration. Using this method, choosing instances to keep as-is isn’t available, and the method is automatically chosen as <strong>Run once</strong>. When a capacity task JSON file is uploaded, the resulting plan is displayed in the following text box and can be amended if needed. Alternatively, rather than uploading the file, the contents can be directly pasted into the text box. Choosing <strong>Next</strong> moves to the review screen where the remainder of the process continues in line with using the interactive capacity configurator UI.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-10-Upload-a-Capacity-Configuration-using-JSON.png"><img alt="Figure 10 - Upload a Capacity Configuration using JSON" class="size-full wp-image-23282 aligncenter" height="1087" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2025/02/19/Figure-10-Upload-a-Capacity-Configuration-using-JSON.png" width="1363" /></a></p> 
<p style="text-align: center;"><em>Figure 10 – Upload a Capacity Configuration using JSON</em></p> 
<p>You may also prefer using the AWS CLI/Outposts API for creating capacity tasks, and a number of new CLI/API actions are now available to support this:</p> 
<ul> 
 <li><code>cancel-capacity-task</code> / CancelCapacityTask</li> 
 <li><code>get-capacity-task</code> / GetCapacityTask</li> 
 <li><code>list-blocking-instances-for-capacity-task</code> / ListBlockingInstancesForCapacityTask</li> 
 <li><code>list-capacity-tasks</code> / ListCapacityTasks</li> 
 <li><code>start-capacity-task</code> / StartCapacityTask</li> 
</ul> 
<p>In addition to the same options available within the console, it is possible to request a dry run of the capacity task to determine if the instance type and instance size changes are above or below the available instance capacity. Requesting a dry run doesn’t make any changes to your plan.</p> 
<p>For example, using the CLI to submit a capacity task to homogeneously slot an Outpost with 2 x m5 and 2 x c5 hosts (192 vCPU for each capacity pool) with xlarge instance sizes, and using periodic running could be achieved by running the following:</p> 
<p><code>aws outposts start-capacity-task \</code></p> 
<p><code>--outpost-identifier <em>outpost-arn</em> \</code></p> 
<p><code>--instance-pools '[{"InstanceType":"c5.xlarge","Count":48},{"InstanceType":"m5.xlarge","Count":48}]' \</code></p> 
<p><code>--task-action-on-blocking-instances WAIT_FOR_EVACUATION</code></p>
