<p><i>This post is written by Kate Sposato, Senior Solutions Architect, EC2 Edge Compute<br /> </i></p> 
<p>AWS is excited to announce deeper collaboration with industry-leading storage solutions to streamline the use of third-party storage with <a href="https://aws.amazon.com/outposts/">AWS Outposts</a>. You can now attach and use external block data volumes from NetApp® on-premises enterprise storage arrays and Pure Storage® FlashArray<img alt="™" class="wp-smiley" src="https://s.w.org/images/core/emoji/14.0.0/72x72/2122.png" style="height: 1em;" /> directly from the <a href="https://aws.amazon.com/console/">AWS Management Console</a>.</p> 
<p>Outposts is a fully managed service that extends AWS infrastructure, AWS services, APIs, and tools to customer premises. By providing local access to AWS managed infrastructure, Outposts allows you to build and run applications on premises using the same application programming interfaces (APIs) as in <a href="https://aws.amazon.com/about-aws/global-infrastructure/regions_az/">AWS Regions</a>. Moreover, this is done while using local compute and storage resources to meet lower latency and local data processing needs. Outposts is available in various rack and server form factors.</p> 
<p>Many of you have block storage systems running in your on-premises environments that provide advanced data storage and management features—such as snapshots, replication, and encryption—to protect data integrity and security. There are various uses cases that would predicate you needing to access data through these external volumes backed by external storage systems from an application running in <a href="https://aws.amazon.com/ec2/">Amazon Elastic Compute Cloud (Amazon EC2)</a> instances on Outposts. These include: regulatory auditing requirements, government and local regulation compliance, high data durability and resiliency requirements, low-latency data access, and migration of on-premises applications that are tightly coupled with existing external storage systems. To make it easier for you to use external volumes with Outposts, AWS has validated a broad range of third-party storage solutions through the <a href="https://aws.amazon.com/outposts/partners/">AWS Outposts Ready Program</a>. With this program, you can easily identify <a href="https://partners.amazonaws.com/search/partners/?facets=Use%20Case%20%3A%20Storage%7CUse%20Case%20%3A%20Storage%20%3A%20Backup%20and%20Restore%7CUse%20Case%20%3A%20Storage%20%3A%20Primary%20Storage%7CUse%20Case%20%3A%20Storage%20%3A%20Archive%7CUse%20Case%20%3A%20Storage%20%3A%20Business%20Continuity%2FDisaster%20Recovery%7CProduct%20%3A%20AWS%20Outposts">storage solutions that are tested to run with Outposts</a>.</p> 
<p>Today, we are taking our integration with storage solutions from NetApp and Pure Storage to the next level. Outposts now has a simplified and automated way to launch EC2 instances with attached block storage from external infrastructure through the AWS Management Console. The new integration includes automated user script generation and attachment of data volumes to EC2 instances running on 42U Outposts racks and 2U Outposts servers. This integration reduces the friction associated with using the advanced data management and security features of external storage infrastructure in combination with Outposts, allowing you to create a resilient, compliant, and optimized storage and compute infrastructure.</p> 
<h2>Outposts rack storage and networking overview</h2> 
<p>Outposts racks support <a href="https://aws.amazon.com/ebs/">Amazon Elastic Block Store (Amazon EBS)</a> volumes for EC2 instances, which provide persistent local block storage.</p> 
<p>EC2 instances running on Outposts racks can access data stored on external block storage arrays over the Outposts <a href="https://docs.aws.amazon.com/outposts/latest/userguide/outposts-local-gateways.html">local gateway (LGW)</a>. An LGW enables connectivity between the Outpost subnets, where EC2 instances run, and the on-premises network. It carries storage traffic between the EC2 instances running on the Outposts rack and the local network. The LGW is created by AWS as part of the Outposts rack installation process. Each Outposts rack supports a single LGW.</p> 
<p>The following diagram shows an EC2 instance running on an Outposts rack with an <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/using-eni.html">elastic network interface (ENI)</a> and LGW configured for instance connectivity. An external storage array communicates with the EC2 instance running on the Outposts rack through the <a href="https://docs.aws.amazon.com/whitepapers/latest/aws-outposts-high-availability-design/network-attachment.html">Outpost network devices (ONDs)</a>. <a href="https://docs.aws.amazon.com/whitepapers/latest/aws-outposts-high-availability-design/network-attachment.html">Customer Network Devices (CNDs)</a> that connect to EC2 instances running on Outposts racks need to support the following:</p> 
<ul> 
 <li>Link aggregation: connections to the Outposts rack network devices are added to a link aggregation group (LAG).</li> 
 <li>VLANs: Virtual LANs (VLANs) are configured between each Outposts rack TOR device and any customer devices, including data stores.;</li> 
 <li>Dynamic routing: Border Gateway Protocol (BGP) is configured between the CND and the OND for each VLAN. Two total BGP sessions are shown in the following diagram between devices.</li> 
</ul>

<h2>Outposts server storage and networking overview</h2> 
<p>Outposts servers come with internal NVMe SSD-based high-performance <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/InstanceStorage.html">instance storage</a>. Similar to AWS Regions, instance storage is allocated directly to the EC2 instance and follows the lifecycle of the instance. For example, if an EC2 instance is terminated, then the instance storage associated with the instance is also deleted. If you want data to persist after the instance is terminated, you can use external storage solutions to complement the instance storage included with Outposts servers.</p> 
<p>Outposts servers have a <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/local-network-interface.html">local network interface (LNI)</a>. This logical networking component connects the EC2 instances running on the Outposts servers subnet to the on-premises network and allows communication to other on-premises storage, compute, and networking appliances.</p> 
<p>To support the Amazon EC2 on Outposts to external storage array integration, an LNI must be created then added to the EC2 instance during instance launch. An LNI can only be created through the <a href="https://aws.amazon.com/cli/">AWS Command Line Interface (AWS CLI)</a> or the <a href="https://aws.amazon.com/developer/tools/">AWS software development toolkit (SDK)</a> using the following command. The <code><span style="color: #ff0000;">subnet id</span></code> is the Outposts server subnet and the <code><span style="color: #ff0000;">device index</span></code> should be unique to the subnet.</p> 
<pre><code>aws ec2 modify-subnet-attribute --subnet-id <span style="color: #ff0000;">&lt;subnet id&gt;</span> --enable-lni-at-device-index <span style="color: #ff0000;">&lt;device index&gt;</span></code></pre> 
<p>In the on-premises network, you must have a Network Interface Card (NIC) at the same device index that you specified when running the preceding CLI command.</p> 
<p>Further detailed steps for this workflow are listed in the <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/add-lni.html">Outposts server user guide</a>.</p> 
<p>When the <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/add-lni.html">local network interfaces are enabled</a> on an Outpost subnet, the EC2 instances in the Outpost subnet can be configured to include this LNI in addition to the ENI. The LNI connects to the on-premises network while the ENI connects to the VPC.</p> 
<p>The following diagram shows an EC2 instance running on an Outposts server with both an ENI and LNI configured for instance connectivity. <h2>Supported operating systems and AWS Support</h2> 
<p>The rest of this post covers the steps for how to launch an EC2 instance running on an Outposts 2U server or Outposts rack with a connected external block storage volume for local data access from within the EC2 instance. The current release of this feature supports EC2 instances running Microsoft Windows Server 2022 and Red Hat Enterprise Linux 9 (RHEL9) based operating systems.</p> 
<p>Support for Outposts and all Outposts integration features, including this one, needs an active <a href="https://aws.amazon.com/premiumsupport/plans/enterprise/">AWS Enterprise Support Plan</a> or <a href="https://aws.amazon.com/premiumsupport/plans/enterprise-onramp/">AWS Enterprise On-Ramp Support Plan</a>. Support for external storage arrays and configurations can be obtained from the respective storage vendor and may need an additional support plan depending on the vendor and the storage solution implemented.</p> 
<p>This post assumes you’re familiar with the basic functionality of Outposts servers and Outposts rack. If you would like to learn more about the Outposts family in general, then the user guide, <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/what-is-outposts.html">What is AWS Outposts?</a>, is a great place to start.</p> 
<h2>Solution deployment</h2> 
<p>The following sections outline the solution deployment.</p> 
<h3>Prerequisites:</h3> 
<ol> 
 <li>An Outposts 2U server or Outposts rack is provisioned, activated, and connected to the customer network.</li> 
 <li>A block storage array is connected on the same network and accessible to Outposts subnets.</li> 
 <li>A block data volume is configured and running on the storage array. The unique identifier for this volume is necessary for launching the EC2 instance on the Outpost. The volume must remain provisioned after initial provisioning on the storage array.</li> 
 <li>The IP address and port number (optional for iSCSI connections) of the block storage volume, which is necessary for launching the EC2 instance on the Outpost.</li> 
</ol>

<h3>Deployment steps for NVMe-over-TCP connections</h3> 
<p>1. (Prerequisite) If there is no block data volume already running and configured on the compatible storage array, this must be completed in the storage solution’s interface before moving to Step 2.</p> 
<p style="padding-left: 40px;">a. Create an NVMe device, subsystem, and namespace for the block data volume.</p> 
<p style="padding-left: 40px;">b. Optionally, generate a host NQN that is used for the EC2 instance connection, and add it to the allow list for the appropriate subsystems.</p> 
<p style="padding-left: 40px;">c. The following pieces of information are used in later steps:</p> 
<p style="padding-left: 80px;">i. Host NQN: Unique identifier of the EC2 instance for attachment;</p> 
<p style="padding-left: 80px;">ii. Target IP: Address of the connected block volume host;</p> 
<p style="padding-left: 80px;">iii. Target Port Number: Port number of the connected block volume host.</p> 
<p>You can learn more about launching and configuring external storage arrays in the <a href="https://docs.aws.amazon.com/outposts/">Outposts family documentation</a> or in the respective storage array vendor documentation.</p> 
<p>2. In the Console, navigate to <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-launch-instance-wizard.html">EC2 Launch Instance Wizard</a> by choosing <strong>EC2, Instances, Launch instances</strong>.</p> 
<p style="padding-left: 40px;">a. Name the instance and add any desired tags to be applied at launch.</p> 
<p style="padding-left: 40px;">b. Choose the desired, compatible RHEL9 based <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/AMIs.html">Amazon Machine Image (AMI)</a> from the list, or choose one from the <a href="https://aws.amazon.com/marketplace">AWS Marketplace</a>.</p> 
<p style="padding-left: 40px;">c. Choose the desired <strong>EC2 Instance type</strong>.</p> 
<p style="padding-left: 40px;">d. Expand the <strong>Network settings</strong> section and select <strong>Edit</strong>. Choose the VPC and subnet of the target Outpost.</p> 
<p style="padding-left: 80px;">i. Outposts servers only: <a href="https://docs.aws.amazon.com/outposts/latest/server-userguide/add-lni.html">You must create an LNI</a> in the Advanced Network settings before launching the instance.</p> 
<p style="padding-left: 40px; text-align: left;">e. Expand <strong>Advanced network configuration</strong> and select <strong>Add network device</strong>. Continue to add network devices until the <strong>Device index</strong> is equal to the volume index.</p> 
<p style="padding-left: 40px; text-align: center;"><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-4.-Advanced-network-configuration.png"><img alt="Figure 4. Advanced network configuration" class="aligncenter wp-image-23189" height="676" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-4.-Advanced-network-configuration.png" width="780" /></a>Figure 4. Advanced network configuration</p> 
<p style="padding-left: 40px; text-align: left;">f. Expand <strong>Configure storage</strong> and select <strong>Edit</strong> next to <strong>External storage volumes</strong> settings section and choose NVMe/TCP in <strong>Storage network protocol</strong>.</p> 
<p style="padding-left: 40px; text-align: center;"><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-5.-External-storage-volumes-configuration.png"><img alt="" class="aligncenter wp-image-23190" height="433" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-5.-External-storage-volumes-configuration.png" width="780" /></a>Figure 5. External storage volumes configuration</p> 
<p style="padding-left: 40px;">g. Enter the HostNQN in the format provided for the NVMe/TCP data volume. Make sure that the HostNQN used has been added to the storage array subsystem allow list.</p> 
<p style="padding-left: 40px;">h. Select <strong>Add NVMe/TCP Discovery Controller</strong> and enter the IP address and port of the controller from the storage array. Enter 4420 as the<strong> Target Port</strong>, if the target port is unknown.</p> 
<p style="padding-left: 40px;">i. (Optional) You can add more data volumes that use a different target discovery controller at this time by choosing the <strong>Add NVMe/TCP Data Volume</strong> button under the <strong>Target IP</strong> address. Repeat Steps 2.h for each data volume to be attached to the EC2 instance.</p> 
<p style="padding-left: 40px;">j. Expand the <strong>Advanced details</strong> and provide any additional Amazon EC2 behavior settings as appropriate.</p> 
<p style="padding-left: 40px; text-align: left;">k. At the bottom of the <strong>Advanced details</strong> section is the automatically generated <strong>User data</strong>. If you need to manually edit this data, you can do so by selecting <strong>Edit</strong> at the bottom.</p> 
<p style="padding-left: 40px; text-align: center;"><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-6.-Automatically-generated-user-data-file.png"><img alt="" class="aligncenter wp-image-23191" height="519" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-6.-Automatically-generated-user-data-file.png" width="480" /></a>Figure 6. Automatically generated user data file</p> 
<p style="padding-left: 40px;">l. When the configurations are set, choose the <strong>Launch instance</strong> button in the right-side column.</p> 
<p>3. The EC2 Launch Instance Wizard now launches an EC2 instance configured as described on the Outpost and attaches the desired external data volume(s) to the EC2 instance.</p> 
<p>4. Applications and users can access the data on the attached external volumes from within the EC2 instance. To verify this:</p> 
<p style="padding-left: 40px;">a. From within the launched EC2 instance, run <code>sudo nvme list</code></p> 
<p style="padding-left: 40px;">b. The volumes are displayed as <code>/dev/nvme<span style="color: #ff0000;">1</span>n1</code> with the number increasing for each attached volume. Local instance store volumes on Outposts servers and EBS boot volumes on Outposts racks are listed first. External volumes are listed after those with sequentially increasing node numbers.</p> 
<p>5. External storage volume and array management, configuration, and backups continue to be managed through the storage vendor-provided toolkit. You can find more information on external storage management in the respective storage array vendor documentation.</p> 
<h3>Deployment steps for iSCSI connections</h3> 
<p>1. (Prerequisite) If there is no block data volume already running and configured on the compatible storage array, this must be completed in the storage solution’s interface before moving to Step 2.</p> 
<p style="padding-left: 40px;">a. Create an Initiator group (igroup) and add the Initiator IQN to the igroup. Then map the logical unit number (LUN) to the igroup.</p> 
<p style="padding-left: 40px;">b. Optionally, generate an initiator IQN that is used for the EC2 instance connection, and add it to the allow list for the appropriate subsystems.</p> 
<p style="padding-left: 40px;">c. The following pieces of information are used in later steps:</p> 
<p style="padding-left: 80px;">i. Initiator IQN: Unique identifier of the EC2 instance for attachment;</p> 
<p style="padding-left: 80px;">ii. Target IQNs: Unique identifier of the storage virtual machine (SVM);</p> 
<p style="padding-left: 80px;">iii. Target IP: Address of the connected block volume host;</p> 
<p style="padding-left: 80px;">iv. (Optional) Target Port Number: Port number of the connected block volume host.</p> 
<p>You can learn more about launching and configuring external storage arrays in the <a href="https://docs.aws.amazon.com/outposts/">Outposts family documentation</a> or in the respective storage array vendor documentation.</p> 
<p>2. In the Console, navigate to EC2 Launch Instance Wizard by choosing <strong>EC2, Instances, Launch instances</strong>.</p> 
<p style="padding-left: 40px;">a. Name the instance and add any desired tags to be applied at launch.</p> 
<p style="padding-left: 40px;">b. Choose the desired, compatible RHEL9 or Windows Server 2022 based AMI from the list, or purchase one from the AWS Marketplace.</p> 
<p style="padding-left: 40px;">c. Choose the desired <strong>EC2 Instance type</strong>.</p> 
<p style="padding-left: 40px;">d. Expand the <strong>Network settings</strong> section and choose the <strong>VPC</strong> and <strong>subnet</strong> of the target Outpost.</p> 
<p style="padding-left: 80px;">i. Outposts servers only: You must create an LNI in the Advanced Network settings before launching the instance.</p> 
<p style="padding-left: 40px;">e. Expand <strong>Advanced network configuration</strong> and select <strong>Add network device</strong>. Continue to add network devices until the <strong>Device index</strong> is equal to the volume index.</p> 
<p style="text-align: center;"><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-7.-Advanced-network-configuration.png"><img alt="Figure 7. Advanced network configuration</p> 
<p style="padding-left: 40px; text-align: left;">f. Expand <strong>Configure storage</strong> and select <strong>Edit</strong> next to <strong>External storage volumes</strong> settings section and choose <strong>iSCSI</strong> in <strong>Storage network protocol</strong>.</p> 
<p style="padding-left: 40px; text-align: center;"><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2024/11/30/Figure-8.-External-storage-volumes-configuration.png"><img alt="Figure 8. <p style="padding-left: 40px;">g. Enter the <strong>Initiator IQN</strong> for the iSCSI data volume in the format provided. Make sure that the Initiator IQN used has been added to the allow list for the volume.</p> 
<p style="padding-left: 40px;">h. Select <strong>Add iSCSI Target</strong> and enter the Target IP, Target Port, and Target IQN of the storage array. Enter 4420 for the <strong>Target Port</strong>, if the target port is unknown.</p>
